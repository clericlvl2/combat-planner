# ADR-003 — Local persistence: IndexedDB via Dexie

Status: **Accepted** (2026-06-28).

**Decision.** Persist all `AppData` in IndexedDB through Dexie. Combats are individual records (settings a singleton record), so single-combat export/import maps to one row.

**Rationale.** IndexedDB is **transactional** (an interrupted write must not corrupt existing combats), **async** (no main-thread jank on writes at the 30×100 caps), and roomy (no ~5 MB localStorage ceiling). Dexie is a thin, typed wrapper with clean transaction + bulk APIs.

**Alternatives.** `idb` (same engine, less sugar — viable, chosen against only for Dexie's ergonomics); localStorage whole-blob (no transactions → corruption risk, synchronous main-thread blocking, size cap — rejected).

**Consequences.** A `dataVersion` marker is stored for safe import + future migrations. Writes are debounced/batched per action. The store layer (ADR-002) is the only writer. Each combatant's `hpLog` is stored inline on the combatant within the combat record, so it persists across reload/update and **rides along with single-combat export/import** automatically (deliberately unlike the undo/redo stack, which is stripped on export) — no separate table or join needed.
